OpenNebula — Nền Tảng Cloud

CLOUD PLATFORM

OpenNebula 6.4 là nền tảng quản lý cloud. Nó điều phối việc tạo, chạy, di chuyển và xóa VM trên các KVM host. Tương tự vCenter cho VMware, nhưng mã nguồn mở.

OpenNebula là gì?

OpenNebula là Cloud Management Platform mã nguồn mở — tương tự VMware vCenter nhưng cho KVM. Nó cung cấp một lớp điều phối thống nhất để quản lý toàn bộ lifecycle VM trên nhiều KVM host vật lý.

  • ▶ VM Lifecycle — Tạo, start, stop, suspend, migrate, delete VM từ một giao diện duy nhất (UI hoặc CLI)
  • ▶ High Availability — 3-node HA: 1 primary (oned chạy) + 2 standby. DB MariaDB replicated. Failover tự động khi node chính chết
  • ▶ Live Migration — Di chuyển VM đang chạy giữa các KVM host không downtime (copy RAM state qua mạng)
  • ▶ Multi-tenancy — Quản lý Users, Groups, Quotas (giới hạn CPU/RAM/disk mỗi tenant)
  • ▶ Storage Integration — Kết nối với Ceph RBD, NFS, local storage cho VM disk images
  • ▶ Network Management — Quản lý Virtual Networks, IP pools, VLAN, bridge configuration

OpenNebula — Tạo & Quản Lý VM

VM OPERATIONS

Cách 1: Tạo VM qua Sunstone (Web UI)

  1. Truy cập Sunstone tại http://103.142.137.2:9869, đăng nhập bằng tài khoản admin hoặc user
  2. Vào menu Instances → VMs, nhấn nút Create ở góc trên phải
  3. Chọn Template phù hợp (Ubuntu, CentOS, Windows...)
  4. Điều chỉnh cấu hình nếu cần: số CPU, dung lượng RAM, kích thước Disk
  5. Chọn Virtual Network — hệ thống tự động cấp IP từ pool
  6. Nhấn Create — Scheduler tự chọn KVM host phù hợp và khởi động VM
  7. VM xuất hiện trong danh sách với trạng thái RUNNING sau 30-60 giây

Cách 2: Tạo VM qua CLI

SSH vào turbo0201 với user oneadmin, dùng lệnh onetemplate instantiate với tham số tên VM, số CPU và RAM. Có thể tạo nhiều VM cùng lúc hoặc theo batch.

Tự động giám sát: Khi VM được tạo, libvirt_exporter trên KVM host tự detect và expose metrics. Không cần cấu hình thêm gì trong Prometheus.

KVM & Ảo Hóa

VIRTUALIZATION

KVM (Kernel-based Virtual Machine) là hypervisor tích hợp trong Linux kernel. Mỗi VM là một process QEMU-KVM với hardware virtualization (VT-x/VT-d).

KVM là gì?

  • ▶ Type-1 Hypervisor — KVM biến Linux kernel thành bare-metal hypervisor. Không có lớp phần mềm trung gian giữa VM và phần cứng.
  • ▶ Hardware-assisted — Dùng CPU extensions VT-x (Intel) / AMD-V (AMD) để chạy VM với hiệu suất gần native.
  • ▶ QEMU + KVM — QEMU giả lập thiết bị (disk, NIC, USB). KVM cung cấp CPU/RAM virtualization qua kernel. Hai thứ kết hợp = một process cho mỗi VM.
  • ▶ VirtIO drivers — VM dùng VirtIO drivers để đạt hiệu suất disk/network tốt hơn nhiều (gần native I/O).
  • ▶ Ai dùng KVM? — Google Cloud, DigitalOcean, IBM Cloud, Red Hat OpenShift đều chạy trên KVM.

Prometheus & Exporters

MONITORING CORE

Prometheus là gì?

Prometheus là time-series database dùng mô hình Pull: nó chủ động kết nối đến từng exporter theo lịch (mặc định mỗi 15 giây), kéo về toàn bộ metrics, và lưu vào bộ lưu trữ nội bộ (TSDB). Dữ liệu được giữ 90 ngày, giới hạn 50GB.

  • ▶ Scrape interval: 15 giây cho node/libvirt exporter, 30 giây cho SNMP
  • ▶ Alert rules: đánh giá mỗi 15 giây, gửi alert khi điều kiện đúng liên tục trong thời gian "for"
  • ▶ Khoảng 22 scrape jobs tạo ra hơn 100,000 time series
  • ▶ Config tại: /opt/monitoring/prometheus/prometheus.yml

CrowdSec & YARA

IDS + MALWARE

CrowdSec là gì?

CrowdSec là hệ thống phát hiện hành vi tấn công dựa trên cộng đồng. Nó phân tích log (auth.log, audit.log, syslog) để nhận diện các pattern như SSH brute force, port scan, exploit attempts. Hiện tại đang ở chế độ DETECT-ONLY — chỉ phát hiện, không tự block.

  • ▶ 18 agents (17 KVM hosts + monitoring server)
  • ▶ Central LAPI tại monitoring server tổng hợp alerts
  • ▶ Bouncer tắt — Phase 5 sẽ bật sau quan sát đủ

YARA là gì?

YARA là công cụ nhận dạng malware dựa trên pattern matching. Nó so sánh file với 523 rule files để phát hiện malware, ransomware, trojan. Chế độ scan-only — không tự xóa hay quarantine.

Quy trình khi phát hiện IP tấn công

  1. Xem alert CrowdSec trong Grafana dashboard hoặc Telegram
  2. Kiểm tra IP đang tấn công: xem loại tấn công và số lần trong CrowdSec dashboard
  3. Xác định IP đó là external hay internal (nếu internal — có thể là host bị compromise)
  4. Nếu là false positive (IP nội bộ hợp lệ): thêm vào whitelist
  5. Nếu là tấn công thật: ghi nhận IP, thông báo team, xem xét block thủ công qua firewall
  6. Theo dõi tiếp trong 24h để xác nhận tấn công đã dừng

Ceph Storage

DISTRIBUTED STORAGE

Ceph hoạt động thế nào?

  • ▶ 3x Replication: Mỗi block dữ liệu được ghi lên 3 OSD khác nhau. Mất 1 disk → không mất data
  • ▶ CRUSH Algorithm: Client tự tính toán vị trí data (không qua server trung gian), giảm bottleneck
  • ▶ Self-healing: Khi OSD chết, Ceph tự phân phối lại data lên OSD khác (backfill)
  • ▶ MON Quorum: Cần ít nhất 2/3 MON nodes hoạt động để cluster tiếp tục nhận I/O
  • ▶ Podman (quan trọng!): Ceph hosts dùng podman, không phải docker. Lệnh docker sẽ không hoạt động

Quy trình khi disk/OSD lỗi

  1. Nhận alert CephUnhealthy từ Prometheus/Telegram
  2. SSH vào MON node (turbo0201 hoặc 0207), chạy lệnh kiểm tra Ceph status trong container podman
  3. Xem OSD nào bị down và disk vật lý nào tương ứng
  4. Kiểm tra disk vật lý bằng công cụ SMART — xác định disk hỏng hay chỉ bị disconnect
  5. Nếu disk hỏng: thay disk mới, Ceph tự backfill (có thể mất 1-4 giờ tùy kích thước)
  6. Theo dõi tiến trình recovery qua Grafana dashboard cho đến khi HEALTH_OK

Quy trình cấp thêm storage cho khách

  1. Kiểm tra Ceph pool còn đủ dung lượng trống (tối thiểu 20% free)
  2. Vào OpenNebula Sunstone — chọn VM khách
  3. Tắt VM (Poweroff), vào tab Storage → Disk → Resize hoặc Add new disk
  4. Khởi động lại VM, SSH vào VM và mở rộng partition/filesystem

Lưu ý: Standby MGR không expose Prometheus metrics — đây là hành vi bình thường. Chỉ Active MGR mới expose. Đừng báo lỗi nếu thấy 1/2 Ceph MGR targets down trong Prometheus.

Thêm / Xóa Host

HOST MANAGEMENT

Quy trình thêm KVM host mới vào monitoring

  1. Cài node_exporter và libvirt_exporter trên host mới bằng script cài đặt tương ứng với OS (Oracle Linux 8 hoặc CentOS 7). Kiểm tra exporter đã respond trên port 9100 và 9177.
  2. Cấu hình firewall trên host mới: chỉ cho phép monitoring server (103.142.137.6) kết nối vào port 9100 và 9177. Từ chối tất cả IP khác.
  3. Cài Wazuh Agent trên host mới bằng script deploy-wazuh-agents.sh. Xác nhận agent đã connect và xuất hiện trong Wazuh Dashboard.
  4. Cài CrowdSec và YARA bằng script setup-crowdsec.sh và setup-yara.sh. Kiểm tra CrowdSec đã đăng ký với Central LAPI.
  5. Cài audit rules bằng script setup-audit-rules.sh để Wazuh có dữ liệu audit đầy đủ.
  6. Cập nhật prometheus.yml trên monitoring server: thêm IP:9100 và IP:9177 của host mới vào đúng job (cluster1 hoặc cluster2).
  7. Reload Prometheus config — không cần restart, chỉ cần gọi API reload để áp dụng ngay.
  8. Mở Prometheus Targets tại :9090/targets — xác nhận host mới có trạng thái UP.

Quy trình xóa host khỏi monitoring

  1. Trước tiên: đảm bảo tất cả VM đã được migrate sang host khác. Kiểm tra danh sách VM đang chạy trên host trong OpenNebula.
  2. Xóa host khỏi Prometheus config: comment hoặc xóa IP:9100 và IP:9177 khỏi prometheus.yml, reload config.
  3. Xóa Wazuh Agent: dùng lệnh manage_agents trong wazuh.manager container với Agent ID tương ứng.
  4. Xóa CrowdSec machine: dùng cscli machines delete với hostname của host.
  5. Nếu cần: SSH vào host và dừng tất cả monitoring services (node_exporter, libvirt_exporter, wazuh-agent, crowdsec) rồi disable chúng.
